# AI Java 面试训练与简历优化平台 PRD - V1

## 1. 文档说明

本文档为“AI Java 面试训练与简历优化平台”的 V1 版本产品需求文档。V1 版本目标是完成项目的最小可用闭环，重点实现用户登录、题库基础管理、简历录入、AI 模拟面试、动态追问、面试记录和面试报告。

该项目定位为个人作品集项目，用于展示 Java 后端、Vue 前端、AI 应用接入、业务流程设计和基础工程化能力。项目不包装成真实企业系统，不声明已有企业客户或真实商业数据。

## 2. 项目背景

传统 Java 面试准备工具通常以静态题库为主，用户只能查看题目和答案，缺少真实面试中的连续追问、项目深挖和个性化反馈。真实 Java 面试不仅会问八股文，还会根据候选人的简历项目、技术栈、目标岗位和行业背景进行深入追问。

本项目希望通过 AI 模拟面试官，让用户在正式面试前进行更接近真实场景的训练。系统将结合题库、用户简历、目标岗位和面试配置，生成面试大纲，并在面试过程中根据用户回答进行点评和追问，最终生成结构化面试报告，帮助用户发现知识薄弱点和项目表达问题。

## 3. V1 版本目标

V1 版本的核心目标是跑通“AI 模拟面试闭环”。用户可以注册登录，维护个人简历，选择面试类型，进入 AI 模拟面试，回答 AI 面试官的问题，系统根据回答进行动态追问，面试结束后生成面试报告。

V1 版本不追求功能大而全，而是优先保证主流程完整、可演示、可扩展。后续 V2、V3 版本将在 V1 基础上继续加入题库语义去重、AI 简历优化、学习计划、RabbitMQ 异步任务、MinIO 文件存储、SSE 流式输出、Elasticsearch 搜索、数据看板等能力。

## 4. 目标用户

### 4.1 普通用户

普通用户是准备 Java 面试的求职者，主要包括 Java 后端开发、Java 全栈开发、Java 微服务开发方向的求职者。V1 版本重点面向一到三年经验左右的 Java 开发者。

普通用户可以使用题库刷题、维护简历、创建 AI 模拟面试、查看历史面试记录和面试报告。

### 4.2 管理员

管理员负责维护系统基础数据，包括题库、题目分类、题目标签、问题组、Prompt 模板和 AI 调用日志查看。管理员不参与用户面试过程，只负责后台管理和系统配置。

## 5. V1 功能范围

### 5.1 V1 包含功能

V1 版本包含用户登录注册、用户信息管理、管理员后台、题库管理、题目分类管理、题目标签管理、问题组管理、基础刷题、错题记录、收藏题目、简历手动录入、AI 模拟面试、面试大纲生成、动态追问、面试记录、面试报告、Prompt 模板管理和 AI 调用日志。

### 5.2 V1 暂不包含功能

V1 版本暂不支持支付、会员、企业端、招聘投递、真实招聘数据、移动端 App、语音面试、视频面试、简历文件上传解析、MinIO 文件存储、RabbitMQ 异步任务、Elasticsearch 搜索、复杂数据看板、AI 题目批量生成、学习计划自动生成和题库 Embedding 语义去重。

这些功能将放在后续版本中迭代。

## 6. 核心业务闭环

V1 版本的核心业务闭环如下：用户注册登录后，先在系统中维护一份个人简历，包括基本信息、技能栈、工作经历和项目经历。然后用户选择面试类型、目标岗位、经验年限、行业方向、面试难度和面试官风格。系统根据配置和简历内容生成面试大纲。用户进入面试房间后，AI 面试官按照面试大纲逐轮提问。用户回答后，AI 对回答进行评分和点评，并决定是否追问。面试结束后，系统生成面试报告，展示总分、模块得分、问题回顾、薄弱知识点、项目表达问题和复习建议。

## 7. 功能模块设计

## 7.1 用户与权限模块

### 7.1.1 功能说明

用户与权限模块负责系统的注册、登录、认证、用户信息维护和角色权限控制。V1 版本角色分为普通用户和管理员。

### 7.1.2 功能点

普通用户可以注册账号、登录系统、退出登录、修改个人信息、修改密码和查看个人学习概览。管理员可以登录后台，维护题库、分类、标签、问题组和 Prompt 模板。

### 7.1.3 权限要求

普通用户只能访问自己的简历、面试记录、答题记录和面试报告。管理员可以访问后台管理功能，但不应随意修改用户的个人面试记录。

### 7.1.4 技术建议

V1 推荐使用 Sa-Token 实现登录认证和角色权限控制。登录成功后返回 token，前端通过请求头携带 token 访问后端接口。

## 7.2 题库管理模块

### 7.2.1 功能说明

题库管理模块用于维护 Java 面试题基础数据。V1 版本题库既服务于普通刷题，也服务于 AI 模拟面试的问题来源。

### 7.2.2 题目分类

V1 版本建议内置或支持维护以下分类：Java 基础、集合框架、多线程、JVM、Spring、Spring Boot、MyBatis、MySQL、Redis、RabbitMQ、微服务、分布式、设计模式、项目场景题。

### 7.2.3 题目字段

每道题应包含题目标题、题目内容、参考答案、答案解析、题目分类、题目标签、难度等级、题型、适合经验年限、是否高频、状态、所属问题组。

### 7.2.4 问题组设计

V1 版本需要引入问题组 question_group 的概念，用于解决同一考察意图下不同问法的问题归并。例如“说一下 HashMap”“介绍一下 HashMap”“HashMap 的底层原理是什么”可以归入同一个问题组。

V1 版本的问题组归并可以先由管理员手动维护，暂不做 Embedding 语义去重。用户刷题和 AI 面试抽题时，同一场练习或同一场面试中应避免重复出现同一问题组下的题目。

## 7.3 刷题练习模块

### 7.3.1 功能说明

刷题练习模块用于用户日常练习 Java 面试题。V1 版本不需要做复杂刷题模式，但需要支持基础题目浏览和练习记录。

### 7.3.2 功能点

用户可以按分类、标签、难度查看题目列表。用户可以进入题目详情页查看题目内容、提交自己的回答、查看参考答案和解析。用户可以收藏题目，也可以将题目标记为已掌握、模糊、不会。

系统需要记录用户答题记录，包括用户答案、是否查看答案、掌握状态、答题时间和题目所属知识点。

## 7.4 错题本与收藏模块

### 7.4.1 功能说明

错题本与收藏模块用于沉淀用户的薄弱题目和重点题目。

### 7.4.2 功能点

用户可以查看自己的错题列表、收藏列表和未掌握题目列表。用户可以从错题本进入题目详情重新练习，也可以将错题标记为已掌握。

V1 版本的错题可以通过用户手动标记“不会”或 AI 面试报告中的薄弱题目自动加入。

## 7.5 简历管理模块

### 7.5.1 功能说明

简历管理模块用于维护用户的个人简历信息。V1 版本先支持手动录入简历，不支持 PDF、Word 文件上传解析。

### 7.5.2 简历内容

简历应包含基本信息、求职方向、技能栈、工作经历、项目经历和教育经历。其中项目经历是 AI 项目深挖面试的核心数据。

### 7.5.3 项目经历字段

每段项目经历建议包含项目名称、项目时间、项目背景、技术栈、个人职责、核心功能、项目难点、优化成果和可补充说明。

### 7.5.4 使用场景

用户创建项目深挖面试或综合模拟面试时，需要选择一份简历。系统根据简历中的项目经历和技术栈生成项目追问方向。

## 7.6 AI 模拟面试模块

### 7.6.1 功能说明

AI 模拟面试模块是 V1 版本的核心模块。系统通过 AI 扮演 Java 面试官，根据用户选择的面试配置、题库内容和简历项目进行多轮提问、点评和追问。

### 7.6.2 面试模式

V1 版本支持三种面试模式：八股文技术面试、简历项目深挖面试和综合模拟面试。

八股文技术面试围绕 Java 基础、集合、多线程、JVM、Spring、MySQL、Redis、中间件和微服务进行提问。简历项目深挖面试围绕用户简历中的项目经历、技术栈、个人职责和项目难点进行追问。综合模拟面试将八股文、项目深挖和场景题结合，模拟更接近真实 Java 面试的流程。

### 7.6.3 面试配置

用户创建面试时需要选择面试模式、目标岗位、经验年限、行业方向、难度等级、面试官风格和是否基于简历。目标岗位可以包括 Java 后端开发、Java 全栈开发、Java 微服务开发。经验年限 V1 可支持一年、三年、五年。行业方向 V1 可支持通用业务、电商、金融支付、在线教育、SaaS。面试官风格 V1 可支持普通、严厉、项目深挖。

### 7.6.4 面试流程

用户创建面试后，系统生成面试会话和面试大纲。面试开始后，AI 每次只提出一个问题。用户提交回答后，AI 对回答进行评分和点评，并判断是否需要追问。如果需要追问，则生成追问问题；如果不需要追问，则进入当前阶段的下一题或切换到下一阶段。面试达到预设轮数或用户主动结束后，系统生成面试报告。

### 7.6.5 面试阶段

V1 版本综合模拟面试建议包含以下阶段：开场、自我介绍、Java 基础、数据库、缓存与中间件、框架基础、项目深挖、场景设计、总结报告。八股文面试可以只包含技术知识阶段。项目深挖面试可以只包含项目背景、技术选型、核心流程、难点追问和扩展性问题。

## 7.7 面试大纲生成模块

### 7.7.1 功能说明

面试大纲生成模块用于控制 AI 面试流程，避免 AI 随机聊天。系统根据面试配置、简历信息和题库分类生成本场面试的大纲。

### 7.7.2 大纲内容

面试大纲应包含阶段名称、阶段顺序、预计问题数量、考察重点、是否需要基于简历、阶段状态和阶段得分。

### 7.7.3 生成方式

V1 版本可以采用规则生成大纲，结合少量 AI 辅助优化。比如用户选择“三年 Java 后端 + 电商 + 综合面试”，系统默认生成 Java 基础、MySQL、Redis、Spring Boot、项目深挖和电商场景题几个阶段。

## 7.8 动态追问模块

### 7.8.1 功能说明

动态追问模块用于模拟真实面试官根据用户回答继续追问的过程。

### 7.8.2 追问规则

AI 对用户回答进行评分后，根据评分和回答内容决定下一步。如果回答明显错误，应指出关键问题并追问基础概念。如果回答较浅，应追问原理或项目落地细节。如果回答较好，应提高难度或进入场景应用。如果回答涉及简历项目，应优先追问项目中的真实使用方式。

### 7.8.3 追问限制

为了避免面试卡在一个问题上，V1 版本建议每个主问题最多追问两次。超过追问次数后，应进入下一题或下一阶段。

## 7.9 面试记录模块

### 7.9.1 功能说明

面试记录模块用于保存用户的历史面试会话和每轮问答内容。

### 7.9.2 功能点

用户可以查看历史面试列表，列表展示面试模式、目标岗位、行业方向、难度、开始时间、结束时间、总分和状态。用户可以进入面试详情查看完整问答记录，包括 AI 问题、用户回答、AI 点评、得分、追问关系和所属知识点。

## 7.10 面试报告模块

### 7.10.1 功能说明

面试报告模块用于在面试结束后生成结构化反馈。

### 7.10.2 报告内容

面试报告应包含总分、各阶段得分、知识点掌握情况、回答亮点、主要问题、项目表达问题、薄弱知识点、复习建议和推荐练习题。

### 7.10.3 页面展示

报告页面建议展示总分卡片、模块评分、雷达图或柱状图、问题回顾、AI 总结和复习建议。V1 版本可以先使用基础图表，后续版本再增强视觉效果。

## 7.11 Prompt 模板管理模块

### 7.11.1 功能说明

Prompt 模板管理模块用于维护不同 AI 场景下的提示词模板。V1 版本不建议将所有 Prompt 写死在代码里，应支持后台配置。

### 7.11.2 模板类型

V1 版本需要支持八股文提问模板、项目深挖提问模板、回答评分模板、动态追问模板、面试报告生成模板。

### 7.11.3 模板变量

Prompt 模板中应支持变量替换，例如用户目标岗位、经验年限、行业方向、简历内容、当前阶段、当前问题、用户回答、历史问答摘要和题库参考答案。

## 7.12 AI 调用日志模块

### 7.12.1 功能说明

AI 调用日志模块用于记录每次大模型调用，方便排查问题和优化 Prompt。

### 7.12.2 日志字段

日志需要记录用户 ID、调用场景、模型名称、请求 Prompt、响应内容、调用耗时、调用状态、失败原因、创建时间。

### 7.12.3 管理功能

管理员可以在后台查看 AI 调用日志，按用户、场景、状态和时间筛选。V1 版本只需要支持查看和筛选，不需要做复杂统计。

## 8. 页面清单

V1 版本建议包含登录页、注册页、首页工作台、题库列表页、题目详情页、错题本页、收藏题目页、简历列表页、简历编辑页、创建面试页、AI 面试房间页、面试历史页、面试报告页、后台题库管理页、后台分类标签管理页、后台问题组管理页、后台 Prompt 模板管理页和后台 AI 调用日志页。

## 9. 核心流程

## 9.1 创建 AI 模拟面试流程

用户进入创建面试页，选择面试模式、目标岗位、经验年限、行业方向、难度等级和面试官风格。如果选择项目深挖或综合模拟面试，用户需要选择一份简历。系统校验参数后创建 interview_session，并根据配置生成 interview_stage 面试阶段。用户点击开始后进入面试房间。

## 9.2 八股文面试流程

系统根据当前阶段从题库中选择合适题目，优先选择与目标岗位、经验年限、难度匹配的题。AI 根据题目内容、参考答案和面试官风格生成问题。用户提交回答后，AI 根据参考答案和评分规则进行点评，并决定是否追问。系统保存本轮问答记录，并根据结果进入追问或下一题。

## 9.3 简历项目深挖流程

用户选择简历后，系统读取简历中的项目经历和技术栈。系统根据项目技术关键词生成追问方向，例如 Redis、MySQL、RabbitMQ、Spring Boot、微服务等。AI 面试官围绕项目背景、个人职责、技术选型、核心流程、数据库设计、缓存设计、异常处理、性能优化和扩展性进行追问。

## 9.4 面试结束与报告生成流程

用户主动结束面试或系统达到预设轮数后，面试会话状态变更为报告生成中。系统汇总面试阶段、问答记录、评分和 AI 点评，调用报告生成 Prompt，生成面试报告。报告生成成功后，状态变更为已完成，用户可以查看报告。

## 9.5 题目录入与问题组归并流程

管理员新增题目时，需要选择分类、标签、难度和所属问题组。如果已有同类问题组，则将新题归入已有问题组；如果没有，则新建问题组。V1 版本问题组由管理员手动选择，后续版本再加入语义相似度检测。

## 10. 数据表草案

## 10.1 用户与权限相关

sys_user：用户表，存储账号、密码、昵称、头像、邮箱、手机号、状态、创建时间。

sys_role：角色表，存储角色名称和角色编码。

sys_user_role：用户角色关联表。

## 10.2 题库相关

question_category：题目分类表，存储分类名称、父级分类、排序和状态。

question_tag：题目标签表，存储标签名称、状态和创建时间。

question_group：问题组表，存储标准问题标题、标准答案、主知识点、难度、描述和状态。

question：题目表，存储题目标题、题目内容、参考答案、解析、分类、难度、题型、适合经验年限、是否高频、问题组 ID 和状态。

question_tag_relation：题目标签关联表。

user_question_record：用户答题记录表，存储用户 ID、题目 ID、用户答案、掌握状态、是否收藏、是否错题和答题时间。

## 10.3 简历相关

resume：简历表，存储用户 ID、简历名称、求职方向、技能栈、工作经历摘要、教育经历、状态和更新时间。

resume_project：简历项目表，存储简历 ID、项目名称、项目时间、项目背景、技术栈、个人职责、核心功能、项目难点、优化成果。

## 10.4 面试相关

interview_session：面试会话表，存储用户 ID、面试模式、目标岗位、经验年限、行业方向、难度、面试官风格、简历 ID、状态、开始时间、结束时间和总分。

interview_stage：面试阶段表，存储会话 ID、阶段名称、阶段顺序、预计问题数、实际问题数、考察重点、状态和阶段得分。

interview_message：面试消息表，存储会话 ID、阶段 ID、角色、问题内容、用户回答、AI 点评、得分、是否追问、父消息 ID、知识点和创建时间。

interview_report：面试报告表，存储会话 ID、总分、模块评分 JSON、优势总结、问题总结、薄弱知识点、复习建议、推荐题目 JSON 和报告内容。

## 10.5 AI 相关

prompt_template：Prompt 模板表，存储模板名称、模板编码、场景类型、模板内容、变量说明、状态和更新时间。

ai_call_log：AI 调用日志表，存储用户 ID、调用场景、模型名称、请求内容、响应内容、耗时、状态、失败原因和创建时间。

## 11. 接口草案

V1 版本接口可以按模块划分为认证接口、用户接口、题库接口、简历接口、面试接口、报告接口和后台管理接口。

认证接口包括注册、登录、退出、获取当前用户信息。题库接口包括题目列表、题目详情、提交答题记录、收藏题目、错题列表。简历接口包括简历列表、简历详情、新增简历、编辑简历、删除简历、新增项目经历、编辑项目经历。面试接口包括创建面试、生成面试大纲、开始面试、提交回答、获取下一题、结束面试、查看面试历史、查看面试详情。报告接口包括生成报告和查看报告。后台接口包括题目管理、分类管理、标签管理、问题组管理、Prompt 模板管理和 AI 调用日志查询。

## 12. 非功能需求

V1 版本需要保证核心接口响应稳定。普通业务接口建议响应时间控制在 500ms 内。AI 调用类接口允许较长耗时，但需要设置超时时间，建议 30 秒到 60 秒。AI 调用失败时应返回明确错误提示，并记录 ai_call_log。

系统需要对用户数据做基本隔离，普通用户只能访问自己的简历和面试记录。管理员接口必须进行角色校验。密码需要加密存储。

AI 面试过程中应保存每轮问答，避免页面刷新后面试记录丢失。面试会话需要有明确状态，例如未开始、进行中、报告生成中、已完成、已取消、失败。

## 13. 技术栈建议

后端建议使用 Spring Boot 3、MyBatis-Plus、MySQL 8、Redis、Sa-Token、Knife4j、Lombok、MapStruct 或 BeanUtils。V1 版本 Redis 可用于登录态、验证码、题目缓存、AI 调用限流和面试临时上下文缓存，但如果开发时间紧，可以先只使用 MySQL 完成核心闭环。

前端建议使用 Vue3、Vite、TypeScript、Element Plus、Pinia、Vue Router、Axios、ECharts 和 Markdown 渲染组件。

AI 接入建议先封装统一 AIClient，屏蔽具体模型供应商。业务层不要直接调用第三方大模型 API，而是通过 AiService 调用，便于后续切换模型和记录日志。

## 14. V1 验收标准

V1 版本完成后，用户应能完整演示以下流程：注册登录系统，创建或编辑一份简历，管理员维护若干 Java 面试题和问题组，用户创建一场综合模拟面试，AI 根据面试配置和简历进行提问，用户回答后 AI 能进行点评和追问，面试结束后系统生成结构化报告，用户可以在历史记录中查看该场面试和报告。

如果以上流程能够稳定跑通，则 V1 达到验收标准。

## 15. V1 开发优先级

最高优先级是用户登录、简历手动录入、题库基础管理、AI 模拟面试、动态追问、面试记录和面试报告。

中等优先级是错题本、收藏、Prompt 模板管理和 AI 调用日志。

较低优先级是首页数据统计、图表美化、复杂后台筛选和 Redis 深度优化。

## 16. 后续版本预留方向

V2 版本可以重点迭代题库质量和求职辅助能力，包括题库语义去重、AI 简历优化、学习计划、AI 题目生成审核、SSE 流式输出和 MinIO 简历文件上传。

V3 版本可以重点迭代工程化和高级能力，包括 RabbitMQ 异步任务、Elasticsearch 搜索、数据看板、面试报告异步生成、Docker Compose 一键部署、更多行业模板和更完整的 AI 调用监控。

## 2. 项目背景

传统 Java 面试准备方式通常依赖静态题库、面试文章和个人笔记。用户可以看到题目和参考答案，但缺少真实面试中的连续追问、项目深挖、行业场景考察和个性化反馈。

真实 Java 面试并不是简单地问“HashMap 是什么”或“Redis 有哪些数据结构”。面试官通常会根据候选人的回答继续追问，例如：

- 你说 HashMap 底层是数组、链表和红黑树，那什么时候会转红黑树？
- 你说项目中用了 Redis，具体缓存了什么数据？
- 你说使用 RabbitMQ 做异步处理，如何保证消息不丢失？
- 如果你这个项目是电商场景，库存扣减如何避免超卖？
- 如果支付回调重复通知，接口如何保证幂等？

因此，本项目希望构建一个面向 Java 求职者的 AI 面试训练平台，通过题库、简历、行业模板和大模型能力，模拟真实 Java 面试官的提问、追问、点评和报告生成过程，帮助用户发现知识薄弱点、简历项目表达问题和面试回答风险。

本项目定位为个人作品集项目，用于展示 Java 后端、Spring Cloud Alibaba 微服务、Vue 前端、AI 应用接入、题库治理、缓存、消息中间件、搜索、文件存储、接口设计和工程化能力。项目不包装成真实企业招聘系统，不声明真实商业用户、真实企业客户或真实招聘数据。

## 3. 项目目标

### 3.1 产品目标

系统目标是帮助 Java 求职者完成以下闭环：

```text
维护简历与项目经历
  ↓
进行题库刷题和错题沉淀
  ↓
创建 AI 模拟面试
  ↓
AI 按大纲进行八股文、项目深挖和行业场景提问
  ↓
用户回答问题
  ↓
AI 根据回答动态追问、评分和点评
  ↓
面试结束后生成结构化报告
  ↓
根据薄弱点推荐复习方向、题目和学习计划
```

### 3.2 技术目标

项目需要体现以下技术能力：

1. 使用 Spring Cloud Alibaba 设计微服务架构。
2. 使用 Nacos 完成服务注册发现和配置管理。
3. 使用 Spring Cloud Gateway 统一网关入口。
4. 使用 OpenFeign 完成服务间调用。
5. 使用 Sentinel 实现接口限流、熔断和降级。
6. 使用 MyBatis-Plus 操作 MySQL。
7. 使用 Redis 实现缓存、限流、登录态、幂等控制和面试上下文缓存。
8. 使用 Elasticsearch 实现题库、面试记录、简历内容的全文搜索。
9. 使用消息中间件处理 AI 报告生成、简历解析、题目生成、学习计划生成等异步任务。
10. 使用 Vue3 构建用户端和管理端页面。
11. 封装统一 AI 服务层，支持 Prompt 模板、AI 调用日志、超时处理和失败重试。
12. 建立完整的业务流程、状态机、权限控制、异常处理和日志审计机制。

## 4. 项目定位与边界

### 4.1 项目定位

本项目是一个面向个人学习和求职准备的 AI 面试训练工具，主要解决：

- Java 面试题只看答案，缺少真实追问的问题。
- 简历项目写得空泛，不知道面试官会怎么深挖的问题。
- 用户不知道自己在哪些知识点薄弱的问题。
- 用户缺少结构化面试反馈和复习建议的问题。
- 静态题库难以模拟真实 Java 技术面试流程的问题。

### 4.2 项目不做什么

为了避免项目边界失控，本项目明确不做以下内容：

1. 不做真实招聘平台。
2. 不做企业 HR 系统。
3. 不做岗位投递和候选人筛选。
4. 不做付费会员、支付、订单和营销活动。
5. 不做真实企业客户数据。
6. 不做视频面试和语音面试。
7. 不做社交社区。
8. 不做移动端 App。
9. 不声称项目已经在真实企业生产环境大规模使用。

### 4.3 项目展示口径

在简历或面试中，本项目应表述为：

> 个人作品集项目：AI Java 面试训练与简历优化平台。该项目面向 Java 求职者，提供题库刷题、AI 模拟面试、简历项目深挖、动态追问、面试报告、学习计划等功能。后端采用 Spring Cloud Alibaba 微服务架构，结合 MySQL、Redis、Elasticsearch、消息中间件和大模型 API 实现业务闭环。

## 6. 核心业务场景

### 6.1 八股文技术面试

用户选择 Java 后端方向、经验年限、面试难度后，系统从题库中选择 Java 基础、集合、并发、JVM、Spring、MySQL、Redis、消息队列、微服务等方向的问题。

AI 面试官根据题目和用户回答进行评分、点评和动态追问，模拟真实技术面试。

### 6.2 简历项目深挖面试

用户维护简历和项目经历后，AI 根据项目背景、技术栈、个人职责、核心功能、技术难点和优化成果生成可追问点。

AI 面试官围绕项目真实性、业务理解、技术选型、数据库设计、缓存设计、消息队列、异常处理、性能优化和扩展性进行深挖。

### 6.3 行业场景面试

用户选择目标行业，例如电商、金融支付、在线教育、SaaS、内容社区、物流 / ERP、通用后台管理系统。

AI 面试官结合行业模板进行提问，例如：

- 电商：订单、库存、支付、优惠券、秒杀、超卖。
- 金融支付：幂等、对账、金额精度、支付回调、风控。
- SaaS：多租户、数据隔离、套餐限制、企业权限。
- 在线教育：课程、学习进度、考试、防刷、订单支付。

### 6.4 综合模拟面试

综合模拟面试用于模拟更完整的真实 Java 面试流程，包含：

1. 开场与自我介绍。
2. Java 基础。
3. 数据库。
4. 缓存。
5. 框架。
6. 消息队列。
7. 微服务。
8. 简历项目深挖。
9. 行业场景题。
10. 总结评价。

### 6.5 刷题与学习闭环

用户通过题库练习积累答题记录、错题、收藏和掌握状态。系统基于这些数据生成薄弱知识点分析，并在 AI 面试和学习计划中使用。

## 8. 功能需求详细设计

## 8.1 用户与权限模块

### 8.1.1 功能说明

用户与权限模块负责用户注册、登录、认证、授权、角色权限和用户信息管理。

### 8.1.2 功能点

普通用户功能：

- 用户注册。
- 用户登录。
- 用户退出。
- 获取当前登录用户信息。
- 修改昵称。
- 修改头像。
- 修改邮箱。
- 修改手机号。
- 修改密码。
- 查看个人学习概览。
- 查看最近登录时间。

管理员功能：

- 查询用户列表。
- 查询用户详情。
- 启用 / 禁用用户。
- 重置用户密码。
- 分配用户角色。
- 查看用户学习记录。
- 查看用户 AI 调用记录。

### 8.1.3 权限控制

系统采用 RBAC 权限模型。

核心对象：

- 用户。
- 角色。
- 菜单。
- 权限标识。
- 用户角色关系。
- 角色菜单关系。
- 角色接口权限关系。

权限要求：

- 普通用户只能访问自己的简历、答题记录、面试记录和报告。
- 管理员可以访问后台管理模块。
- AI 调用日志、Prompt 模板、系统配置只允许管理员访问。
- 系统接口必须进行登录态校验。
- 管理端接口必须进行角色权限校验。

### 8.1.4 技术建议

- 登录认证：Sa-Token 或 Spring Security + JWT。
- 微服务鉴权：网关统一鉴权 + 服务内注解校验。
- Token 存储：Redis。
- 权限缓存：Redis。
- 前端权限：Vue Router 路由守卫 + Pinia 用户状态。

## 8.4 题库去重与问题组模块

### 8.4.1 功能说明

面试题经常存在“问法不同、考察点相同”的情况，例如：

- 说一下 HashMap。
- 介绍一下 HashMap。
- HashMap 的底层结构是什么？
- HashMap 的底层原理是什么？

这些题不能只根据标题完全一致判断重复，而应该根据“核心考察意图”归并。

因此，系统引入 `question_group` 问题组概念。

### 8.4.2 问题组定义

一个问题组表示一个标准问题或核心考察意图。

示例：

```text
问题组：HashMap 核心原理
标准问题：请介绍 HashMap 的底层结构与核心原理
关联问法：
- 说一下 HashMap
- 介绍一下 HashMap
- HashMap 的底层结构是什么
- HashMap 的底层原理是什么
```

### 8.4.3 问题组字段

| 字段 | 说明 |
|---|---|
| id | 问题组 ID |
| canonical_title | 标准问题标题 |
| canonical_answer | 标准答案 |
| main_category_id | 主分类 |
| main_knowledge_point | 主知识点 |
| difficulty | 难度 |
| experience_level | 适用经验 |
| description | 考察意图说明 |
| status | 状态 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 8.4.4 去重策略

系统支持以下去重策略：

#### 1. 手动分组

管理员新增题目时，可以选择：

- 归入已有问题组。
- 新建问题组。

刷题和 AI 面试抽题时，同一场练习或同一场面试中避免重复抽取同一个问题组下的题目。

#### 2. 文本归一化

系统对题目标题生成 normalized_title。

处理规则包括：

- 去除标点符号。
- 去除多余空格。
- 统一大小写。
- 去除常见语气词。
- 弱化“说一下”“介绍一下”“谈谈”“请解释”等问法。
- 统一中英文符号。
- 对常见技术词做标准化，例如 HashMap、ConcurrentHashMap、Redis。

#### 3. 相似题检测

新增题目时，系统根据 normalized_title、关键词、分类、标签查找疑似重复题。

返回结果包括：

- 疑似重复题标题。
- 所属问题组。
- 相似原因。
- 建议处理方式。

#### 4. AI 考察意图判断

系统可调用 AI 判断两个题目是否属于同一考察意图。

AI 输出结构：

```text
是否同义题：是 / 否
是否建议合并：是 / 否
关系类型：SAME_INTENT / FOLLOW_UP / RELATED / ADVANCED / PREREQUISITE / COMPARE
主要考察点是否一致：是 / 否
建议原因：文本说明
```

#### 5. 人工审核合并

系统不自动删除题目，而是将疑似重复结果展示给管理员，由管理员选择：

- 合并到已有问题组。
- 新建问题组。
- 标记为追问题。
- 标记为相关题。
- 标记为进阶题。
- 忽略。

### 8.4.5 题目关系类型

题目之间支持以下关系：

| 关系类型 | 说明 | 示例 |
|---|---|---|
| SAME_INTENT | 同义题，同一考察意图 | 说一下 HashMap / 介绍一下 HashMap |
| FOLLOW_UP | 追问题 | HashMap 的 put 流程是什么 |
| RELATED | 相关题 | HashMap 为什么线程不安全 |
| ADVANCED | 进阶题 | ConcurrentHashMap 如何保证线程安全 |
| PREREQUISITE | 前置题 | 数组和链表有什么区别 |
| COMPARE | 对比题 | HashMap 和 Hashtable 有什么区别 |

### 8.4.6 面试中的使用方式

AI 模拟面试抽题时：

- 优先按问题组抽取主问题。
- 同一场面试中不重复出现同一问题组。
- 如果用户回答较浅，可以从该问题组的 FOLLOW_UP 或 ADVANCED 关系中选择追问题。
- 如果用户回答较好，可以选择 RELATED 或 COMPARE 题扩展。
- 如果用户回答错误，可以选择 PREREQUISITE 题回退到基础概念。

- 反问式追问。
- 扩展题追问。
- 代码细节追问。
- 业务边界追问。

### 8.12.4 追问控制

后端需要控制：

- 每个问题最大追问次数。
- 每个阶段最大追问次数。
- 总面试轮次。
- 面试时间限制。
- 是否强制进入下一阶段。
- 是否允许用户跳过问题。
- 是否允许用户主动结束面试。

### 8.12.5 回答评分

每轮回答评分维度：

| 维度 | 说明 |
|---|---|
| 准确性 | 是否答对核心概念 |
| 完整性 | 是否覆盖主要考点 |
| 深度 | 是否能讲底层原理 |
| 项目结合度 | 是否能结合真实项目 |
| 表达清晰度 | 是否适合面试表达 |
| 场景应用能力 | 是否能落地到业务场景 |
| 风险点 | 是否存在明显错误或过度包装 |

评分可输出 0-100 分。

### 8.12.6 动态追问示例

用户回答：

```text
Redis 可以用来做缓存，解决数据库压力大的问题。
```

AI 追问：

```text
你说 Redis 用来做缓存，那你在项目中具体缓存的是什么数据？缓存 Key 是怎么设计的？如果数据库数据更新了，你怎么保证缓存和数据库一致？
```

## 11. 微服务架构设计

## 11.1 架构原则

本项目采用 Spring Cloud Alibaba 微服务架构，但仍然保持个人作品集项目的合理边界。服务拆分以业务领域和技术职责为依据，避免为了微服务而微服务。

核心原则：

1. 网关统一入口。
2. 用户认证统一处理。
3. 业务服务按领域拆分。
4. AI 能力独立封装。
5. 文件、搜索、消息任务独立服务化。
6. 服务间调用通过 OpenFeign。
7. 配置和注册通过 Nacos。
8. 限流熔断通过 Sentinel。
9. 跨服务强一致事务尽量减少，优先使用本地事务 + 消息最终一致性。
10. 公共能力抽取为 common 模块。

## 11.5 数据库设计原则

个人作品集项目可以使用同一个 MySQL 实例，不同服务使用不同逻辑库或表前缀。

建议：

- 用户权限库：用户、角色、菜单、权限。
- 题库库：题目、分类、标签、问题组、题目关系。
- 练习库：答题记录、错题、收藏、掌握状态。
- 简历库：简历、项目经历、解析记录、追问点。
- 面试库：面试会话、阶段、消息、报告。
- AI 库：Prompt、模型配置、AI 调用日志。
- 系统库：文件、通知、任务、配置、日志。

## 11.6 分布式事务原则

项目中尽量避免复杂强一致跨服务事务。

建议策略：

- 单服务内使用本地事务。
- 跨服务状态更新使用消息最终一致性。
- 关键幂等使用唯一索引 + Redis 防重复。
- 简历解析、报告生成、AI 题目生成采用任务状态机。
- 必须强一致的场景再考虑 Seata。
- 面试问答记录优先由 interview-service 本地落库，AI 结果失败时记录失败状态并允许重试。

## 15. 消息中间件使用场景

消息中间件用于处理耗时任务和跨服务解耦。

场景包括：

1. 简历文件解析。
2. AI 简历结构化。
3. 面试报告生成。

6. ES 索引同步。
7. 通知发送。
8. AI 调用失败重试。
9. 操作日志异步入库。
10. 文件清理任务。

消息设计原则：

- 消息必须包含业务 ID。
- 消费端必须保证幂等。
- 失败消息记录失败原因。
- 支持最大重试次数。
- 多次失败进入死信队列或失败任务表。
- 任务状态应落库，不能只依赖消息队列。

## 19. 非功能需求

## 19.1 性能要求

- 普通查询接口响应时间建议控制在 500ms 内。
- 管理端复杂查询接口建议控制在 1s 内。
- AI 调用接口允许较长耗时，但应设置超时时间。
- AI 同步调用建议超时时间控制在 30-60s。
- AI 长文本生成建议使用 SSE 或异步任务。
- 搜索接口应支持分页。
- 高频题目、分类、标签应使用缓存。

## 19.2 安全要求

- 密码必须加密存储。
- 用户只能访问自己的简历、面试记录和报告。
- 管理端接口必须校验角色权限。
- 文件下载必须校验权限。
- AI Prompt 中不得泄露用户敏感信息。
- 日志中避免记录明文密码、Token、API Key。
- 系统配置中的 API Key 需要加密或脱敏展示。
- 上传文件需要校验类型和大小。

## 19.3 稳定性要求

- AI 调用失败需要记录失败原因。
- AI 返回格式异常需要兜底处理。
- 异步任务需要状态机。
- 消息消费需要幂等。
- 失败任务支持重试。
- 重试多次失败进入失败任务表或死信队列。
- 面试问答过程需要实时保存，避免刷新页面丢失记录。

## 19.4 可维护性要求

- Prompt 模板支持后台配置。
- AI 调用统一经过 ai-service。
- 公共异常统一处理。
- 统一返回结构。
- 统一错误码。
- 接口文档自动生成。
- 日志带 TraceId。
- 服务间调用需要记录异常。
- 核心业务操作记录操作日志。

## 19.5 可扩展性要求

- AI 模型供应商可替换。
- 消息中间件可替换。
- 文件存储可替换。
- 题库分类可扩展。
- 行业模板可扩展。
- 面试官风格可扩展。

## 21. 项目亮点总结

本项目可提炼为以下作品集亮点：

1. **AI 模拟真实 Java 面试官**  
   支持八股文、简历项目深挖、行业场景、综合模拟面试。

2. **动态追问机制**  
   根据用户回答质量决定追问、纠错、换题或提升难度。

3. **面试大纲与状态机控制**  
   后端控制面试阶段、轮次、追问次数和结束条件，避免 AI 自由聊天。

4. **简历项目深挖**  
   从简历项目中提取技术栈、职责和难点，生成针对性追问。

5. **题库问题组去重**  
   使用 question_group 将不同问法但考察意图相同的题目归并，避免重复刷题和重复面试。

6. **题目关系网络**  
   支持同义题、追问题、相关题、进阶题和前置题，辅助 AI 动态追问。

7. **AI 工程化治理**  
   通过 Prompt 模板、AI 调用日志、限流、超时、失败重试提升稳定性。

8. **Spring Cloud Alibaba 微服务架构**  
   使用 Nacos、Gateway、OpenFeign、Sentinel 等组件展示微服务能力。

9. **Redis 缓存与限流**  
   用于登录态、题目缓存、AI 调用限流、面试上下文和幂等控制。

10. **消息中间件异步任务**  
    用于简历解析、报告生成、AI 题目生成、学习计划生成等耗时任务。

11. **Elasticsearch 搜索能力**  
    支持题库、简历、面试记录和报告全文搜索。

12. **Vue 前端完整展示**  
    包含用户端、管理端、AI 面试房间、面试报告可视化和数据看板。

## 23. 结论

本项目应围绕“AI 模拟真实 Java 面试官”这一核心价值展开，不应退化成普通题库系统，也不应包装成真实招聘平台。

完整项目主线是：

```text
题库治理
  ↓
刷题练习
  ↓
简历管理
  ↓
AI 面试大纲
  ↓
八股文面试
  ↓
项目深挖
  ↓
行业场景追问
  ↓
面试报告
  ↓
学习计划
```

技术架构采用 Spring Cloud Alibaba 微服务体系，结合 MySQL、Redis、Elasticsearch、消息中间件、MyBatis-Plus 和 Vue 前端，能够覆盖 3 年 Java 后端求职中常见的业务建模、微服务、缓存、搜索、异步任务、AI 接入和工程化能力。
